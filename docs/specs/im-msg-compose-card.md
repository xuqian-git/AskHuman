# 需求：IM `/msg` 一次性消息输入卡

> 状态：已实现。  
> 关联现状：`docs/specs/agent-interject.md`、`docs/plans/im-msg-select-card.md`、
> `docs/specs/im-select-card.md`。  
> 实现计划：`docs/plans/im-msg-compose-card.md`。

## 1. 背景

`/msg <内容>` 已支持按关注关系直发，或弹 Agent 选择卡后发送；`/msg <编号> <内容>` 也能直接
发送。但空命令 `/msg` 仍只返回用法文本，`/msg <编号>` 只回显待送达内容。用户必须先在聊天输入
完整命令，缺少与 `/todo`、`/new` 等命令一致的卡片输入入口。

本需求给 `/msg` 增加四渠道原生的一次性消息输入卡。它负责选择目标、查看当前待送达内容、输入一条
新消息并发送；原有带正文的快捷命令和队列语义保持不变。

## 2. 目标与非目标

目标：

1. `/msg` 无参数时可完整走卡片流程，不要求用户手写 Agent 编号和正文。
2. 卡片发送前明确展示目标 Agent 与当前待送达内容，提交后给出准确的“已送达 / 已排队”结果。
3. 飞书、钉钉、Telegram、Slack 语义一致，各自使用已有原生交互能力。
4. 钉钉不新增模板，也不采用 `/todo-auto` 那种同时投放“操作卡 + 新增卡”的双卡形态。
5. 一次提交最多产生一次队列追加；重复回调、过期卡和状态漂移不得误发。

非目标：

- 卡内不提供撤回；继续使用 `/msg-clear <编号>`。
- 输入卡不是常驻管理面板，发送成功后不允许重复发送。
- 不给空闲、结束或 Grok Agent 增加发送能力。
- 不改变 IM 插话的追加、即时送达、排队消费、持久化和“已阅读”异步回执语义。
- 不把普通聊天的下一条文字隐式当作消息；Telegram 只消费对指定 ForceReply 的回复。

## 3. 用户定案

| 编号 | 决策 | 结论 |
|---|---|---|
| D1 | 主流程 | `/msg` 先确定 Agent，再进入消息输入卡 |
| D2 | 唯一关注目标 | 当前渠道恰好关注一个“工作中且非 Grok”的 Agent 时，直接打开该目标的输入卡 |
| D3 | `/msg <编号>` | 目标工作中时打开输入卡；目标空闲时保持现有纯文本回显 |
| D4 | 快捷命令 | `/msg <内容>` 与 `/msg <编号> <内容>` 保持现状，不强制经过输入卡 |
| D5 | 候选范围 | 只列工作中的非 Grok Agent；无候选时回现有无可发送对象提示 |
| D6 | 卡片内容 | 目标摘要、当前待送达条数与正文预览、消息输入框、发送操作 |
| D7 | 撤回 | 不进卡片；保留 `/msg-clear <编号>` |
| D8 | 卡片寿命 | 一次性；发送成功即定格，不支持连续发送 |
| D9 | 成功反馈 | 只在卡内显示目标与“已送达 / 已排队”结果，不显示队列条数、不额外发成功文本 |
| D10 | 阅读回执 | 排队消息被 Agent 消费后，现有“已阅读”回执仍单独发送 |
| D11 | 待送达预览 | 超长时同时保留开头和结尾，明确标注未显示字符数；真实队列不截断 |
| D12 | 钉钉 | 复用现有单选卡模板与提问卡模板，不新增模板 |
| D13 | 钉钉卡片数量 | 允许“先选择、后输入”的顺序两阶段；选择卡定格后才发输入卡，同时只有一张活动卡 |
| D14 | 跨渠道形态 | 允许按平台能力适配，不要求四渠道像素级或消息条数完全一致 |
| D15 | 过期反馈 | 输入卡过期或台账丢失时明确显示“已过期、未发送”，不静默吞掉提交 |
| D16 | 异常重启 | 持久化不含正文的最小卡片身份，保证重启后也能定格旧卡或回应过期 |

## 4. 命令语义

| 输入 | 行为 |
|---|---|
| `/msg` | 唯一关注目标可发送时直接打开输入卡；否则弹工作中 Agent 选择卡，选中后进入输入卡 |
| `/msg <编号>` | 工作中且非 Grok：打开该目标输入卡；空闲：沿用现有待送达纯文本回显；结束、缺失、Grok：沿用现有错误提示 |
| `/msg <内容>` | 保持现状：唯一关注目标时直发，否则带着正文弹目标选择卡，点“发送”后完成 |
| `/msg <编号> <内容>` | 保持现状：校验目标后直接发送 |
| `/msg-clear <编号>` | 保持现状：撤回待送达内容 |

`/msg` 的目标选择继续复用 `msg_options`：只列 `state=working` 且 `kind!=grok`，保留 `[编号]`、
Agent / 项目 / 标题、关注徽标和工作时长。目标选择卡的 `payload` 为空表示“选择后进入输入卡”；带正文
则表示现有的“选择后立即发送”快捷流。两条路径的按钮分别显示“选择”和“发送”。

## 5. 交互流程

```text
/msg
  ├─ 唯一关注且可发送 ─────────────────────┐
  └─ 否 → Agent 选择卡 → 选中并再次校验 ───┤
                                            ↓
                                      一次性消息输入卡
                              （目标 + 待送达预览 + 输入框）
                                            ↓ 提交
                              空输入 → 原卡提示，仍可填写
                              目标漂移 → 定格“未发送”
                              有效输入 → 原子消费卡台账
                                            ↓
                                  `deliver_msg` 追加 / 即时送达
                                            ↓
                              定格“已发送给 [n] · 结果”
```

`/msg <编号>` 命中工作中目标时从“一次性消息输入卡”开始。目标选择和输入之间必须重新读取 Agent
快照；用户提交时再读一次，防止选择后 Agent 已转为空闲或结束。

## 6. 输入卡信息结构

### 6.1 活动态

- 标题：`给 [编号] <Agent> 发送消息` / `Send a message to [n] <Agent>`。
- 目标摘要：标题与项目名；不把 session id 暴露给用户。
- 待送达区：
  - 空队列：灰色“暂无待送达消息”；
  - 非空：`待送达 N 条` + 正文预览；
  - 正文按纯文本 / 安全转义后的等宽或引用样式展示，不能把用户正文当卡片 Markdown 执行；
  - 共享预览预算以 1600 个 Unicode 字符为初始值，超出时保留首尾并插入
    `… 还有 N 字未显示 …`。各渠道若有更低硬限制可继续收紧，但必须保留首尾与省略计数。
- 输入框：多行，单次最多 3000 个 Unicode 字符；placeholder 为“输入要发送给 Agent 的消息”。
- 主操作：`发送` / `Send`。

待送达正文只是打开卡片时的快照。卡片不订阅队列变化；消息被 Agent 消费后仍由现有“已阅读”消息
通知用户。

### 6.2 提交结果

- 空输入：不消费台账、不发送；原卡显示“请输入消息”，允许重新填写。
- 超过 3000 字：不消费台账；原卡显示长度错误。
- 目标不再工作中 / 已结束 / 消失 / 变为不支持：消费台账并定格“该 Agent 已不在工作中，未发送”。
- 有效输入：先原子取走本卡台账，再调用现有 `deliver_msg`，保证重复 callback 不会追加两次。
- 成功终态只显示：`已发送给 [编号] · 已送达`，或 `已发送给 [编号] · 已排队`。
  不回显刚发送的正文，也不另发同步成功文本。

## 7. 四渠道适配

| 渠道 | 目标选择 → 输入 | 输入提交与终态 |
|---|---|---|
| 飞书 | 现有选择卡通过 callback 就地变成自由 JSON 输入卡；直达目标时直接发输入卡 | form input + 主按钮；ACK 直接把原卡更新为校验提示或成功 / 失败终态 |
| Slack | 现有选择消息就地更新为 `plain_text_input` + Send blocks；直达目标时直接发输入消息 | `block_actions` 携带 state；更新同一 message 为提示或终态 |
| 钉钉 | 先用现有单选卡模板；选中后先定格选择卡，再发送一张复用现有提问卡模板的输入卡。两张卡不同时可操作 | 提问模板 `options=[]`、`allow_input=true`；成功后用 `submit_status` / 公私有变量定格。空输入时把 `submitted` 复位为 false 并显示错误 |
| Telegram | 选择卡定格后发送绑定目标的 ForceReply；直达目标时直接发送 ForceReply | 只接受 `reply_to_message_id` 精确命中的非空文本；发送后删除已用完的 ForceReply 提示，并回复用户消息一条短终态，不劫持普通聊天文字 |

钉钉不得为本功能注册新模板，也不得像 `/todo-auto` 一样同时发送“列表操作卡 + 独立新增卡”。选择阶段
完成并定格后才进入输入阶段，因此任一时刻每条 flow 最多只有一张活动卡。

Telegram 不提供“下一条普通消息”兼容路径。用户没有回复指定 ForceReply 时，任何普通文本继续走
`autochannel` 的既有分类与活跃槽逻辑。

## 8. 台账、幂等与过期

- 新增一次性 `MsgCompose` picker；稳定身份仍是 `(channel, message_id)`，内存 payload 只保存目标
  session id、过期时间与恢复态标记，不保存用户输入正文。
- TTL 沿用单选卡的 30 分钟。与普通 picker 不同，另存一份最小恢复台账：`channel`、`message_id`、
  目标 session id、`expires_at`；不得保存待送达预览、用户草稿或提交正文。
- 输入卡创建成功时写恢复台账；发送 / 失败定格 / 到期后删除。异常重启时先把记录恢复为“只可过期、
  不可发送”的 tombstone route，并主动尝试把旧卡定格为“服务已重启，未发送”；成功或到期后删记录。
- 正常 daemon drain / install 时，把未提交输入卡定格为“服务已重启，未发送”。
- 回调复用各渠道现有表单 submit 协议，并以 `(channel, message_id)` 的 `MsgCompose` 台账判型。即使
  daemon 异常重启，恢复 route 也只能定格“已过期，未发送，请重新使用 `/msg`”；Telegram 路由
  丢失时不能误消费普通文字。
- 有效提交必须用“按 message id 原子 take picker”而不是“先 clone、发送后 remove”，阻止客户端重试或
  并发双 callback 造成重复追加。
- 输入正文只在通过服务端长度与目标状态校验后进入 `InterjectStore::append`；卡片渲染和日志不记录正文。

## 9. 降级

- 目标选择卡发送失败：回现有工作中 Agent 文本列表与 `/msg <编号> <内容>` 用法。
- 输入卡发送失败：目标选择卡定格为“无法打开输入卡”，并回一条带目标编号的快捷命令提示；不得把待
  发送正文放进提示（该流程尚未输入正文）。
- 卡片提交后的渠道更新失败不回滚已成功的 `deliver_msg`；异步“已阅读”回执仍按队列记录工作。

## 10. 验收

1. 四渠道 `/msg` 均能从目标选择进入原生输入；唯一关注目标可跳过选择。
2. `/msg <编号>` 对工作中目标打开输入卡，对空闲目标仍回显队列。
3. 输入卡正确展示空队列和非空队列；超长预览保留首尾且真实队列不变。
4. 空输入、超长输入不会消费卡；目标状态漂移不会发送。
5. 成功提交只追加一次并定格卡片，不另发成功文本；排队消息稍后仍有“已阅读”回执。
6. 钉钉全程只使用现有两种模板，选择卡定格后才发送输入卡，不出现两张同时可操作的卡。
7. Telegram 只消费对指定 ForceReply 的回复，普通聊天文字不被劫持。
8. 过期、重复 callback、daemon drain 与异常重启均不误发；最小恢复台账不含任何消息正文，并能让旧卡
   明确定格或在点击时回应“已过期、未发送”。
