# 需求：钉钉文本类附件可靠预览（内联 / 转 docx）

> 状态：已确认（待按计划实现）
> 关联计划：`docs/plans/dingtalk-attachment-preview.md`
> 需求来源：钉钉渠道「文本附件预览」历史待办

## 1. 背景

钉钉机器人只能用 `media/upload` + `sampleFile` 发文件；`sampleFile` 的 `fileType` 仅支持
`doc/docx/xls/xlsx/ppt/pptx/zip/pdf/rar`；且机器人媒体文件**不进钉盘**，不像用户手动上传的钉盘文件
那样可内嵌预览。实测（2026-06-04）：

| 发送方式 | 钉钉表现 |
|---|---|
| `fileType=pdf`（受支持） | 直接预览 / 打开 ✅ |
| `fileType=md`（不支持） | 下载被当成 `.file`，打不开 ❌ |
| `fileType=txt`（UTF-8 无 BOM） | 能打开但乱码（按 GBK 解码）❌ |
| `fileType=txt`（UTF-8 +BOM） | 编码正确但文字白色看不见 ❌ |
| `fileType=html` | 不行 ❌ |

影响：`md / txt / log / html / 代码` 等常见文本类附件（`AskHuman -f` 下发的提问展示附件）无法在钉钉内可靠预览。

## 2. 目标

让钉钉渠道下发的文本类附件能在手机端与桌面端**稳定预览、中文正常、排版美观**，且**不显著增大应用二进制体积、不内置任何字体**。

## 3. 关键调研结论（实测得出，作为方案前提）

- **PDF 路线被否**：纯 Rust 的高保真中文 PDF 只有 typst 可行（输出仅 ~33KB、字体子集化、不内置字体），但 typst 库给二进制带来 **+约 21MB（当前 release 口径）/ +约 15MB（LTO 口径）** 的净增量，与“在意体积”冲突，放弃。
- **DOCX 路线成立**：`fileType=docx` 在钉钉白名单内，可原生预览；中文由钉钉渲染器渲染，**无需内置字体**；docx 仅几 KB，二进制只新增 `pulldown-cmark` + 一个 zip 库（纯 Rust、体积很小）。
- **钉钉 docx 渲染器字体行为**（实测）：
  - **忽略 `docDefaults` 里的字体**，只认**命名段落样式**里的字体。
  - 没有 `Microsoft YaHei / Calibri / Arial(经 docDefaults)`；中文缺省回退到**宋体（衬线）**。
  - 命名样式里指定 **Arial（英文）**、**SimHei（中文黑体）** 可得到无衬线；代码用 **Courier New** 可得到等宽。
- **钉钉内联 markdown（`sampleMarkdown`）渲染能力**（实测）：标题、粗体/斜体、行内代码、列表、表格、**带语言的围栏代码块（含语法高亮）** 均能正常渲染。

## 4. 已确认决策

| 编号 | 决策项 | 结论 |
|---|---|---|
| D1 | 适用范围 | **仅钉钉渠道**对“不可预览的文本类附件”做处理；其它渠道（Telegram 等）仍发**源文件**，行为不变 |
| D2 | 判定方式 | 按**固定扩展名清单**判定是否为文本类（清单见计划）；命中清单且内容为合法 UTF-8 → 进入文本处理；否则按原样发送 |
| D3 | 已可预览类型 | 扩展名属于钉钉白名单（`pdf/doc/docx/xls/xlsx/ppt/pptx/zip/rar`）或图片 → 不处理，按现状发送 |
| D4 | 混合策略 | 保留“短贴正文 / 长转 docx”的混合：短文本**内联**到钉钉消息正文，长文本转 **docx** 文件发送 |
| D5 | 阈值 | **3000 字符**（按解码后内容的字符数）；`≤3000` 视为短，`>3000` 视为长 |
| D6 | 两个开关（设置项，默认均**开**） | ①「小文件内联到正文」：开=短文本内联；②「文本文件转 docx」：开=未内联的文本文件转 docx，关=发源文件 |
| D7 | 分支逻辑 | 文本文件：`短 且 开关①开` → 内联（不再发文件）；否则（长，或开关①关的短文件）→ `开关②开` ? 转 docx : 发源文件 |
| D8 | 内联格式 | `sampleMarkdown` 一条消息：**加粗 header 行**（文件名 + 大小 + 行数）→ **分割线**（`---`）→ 内容；`.md` 用原始 markdown 渲染，非 md 用**带语言标识的围栏代码块**（语言按扩展名映射，便于高亮） |
| D9 | 内联只发消息 | 短文本**只发内联消息**，不再额外发送文件 |
| D10 | docx 命名 | **源文件名 + `.docx`**（如 `report.md` → `report.md.docx`） |
| D11 | docx 渲染（md） | `.md/.markdown` 用 `pulldown-cmark` 解析 → 套用 **GitHub Markdown 样式** 的 docx；**不额外加文件名标题**（内容自带标题结构） |
| D12 | docx 渲染（非 md） | 非 md（代码/txt/json/log…）→ 顶部一行**文件名标题（H1）** + 一个**等宽代码块**（保留缩进/空白） |
| D13 | docx 代码高亮 | docx 代码块**不做语法高亮**（仅等宽），以避免引入语法高亮库带来的体积增大 |
| D14 | docx 字体/样式 | 字体写进**命名样式**：英文 Arial、中文 SimHei、代码 Courier New(英)+SimHei(中)；样式比例与配色套 GitHub Markdown（细节见计划） |
| D15 | md 支持元素 | 标题、段落、粗体/斜体、行内代码、有序/无序列表（含嵌套）、代码块、引用、表格、分隔线、链接（显示文字）；图片**跳过**（显示 alt 文字） |
| D16 | 转换失败兜底 | 读取/转换失败 → 退回发送**源文件**；**不输出任何警告**（静默） |
| D17 | 新增依赖 | `pulldown-cmark`（解析 markdown）+ 一个 zip 库（打包 docx）；均为纯 Rust、体积很小 |
| D18 | 不内置字体 | 全流程**不内置任何字体**，中文渲染交给钉钉端 |

## 5. 约束与既有规则（不可破坏）

- 不引入会显著增大二进制的依赖（typst/语法高亮库等）；不内置字体。
- 不改变其它渠道（Telegram/Popup）行为；不改变 stdout 契约。
- 不改变现有 `-f` 提问附件语义（AI→人、仅展示、不进 stdout）。
- 钉钉发送链路可在 headless（无 GUI）模式运行，方案不得依赖 webview。

## 6. 验收标准

1. 钉钉收到 `-f report.md`（>3000 字符）→ 收到 `report.md.docx`，可在手机/桌面预览，中文无衬线、代码等宽、标题/表格/列表/引用排版正确。
2. 钉钉收到 `-f notes.txt`（≤3000 字符，开关①开）→ 收到一条内联消息：加粗 header（文件名+大小+行数）+ 分割线 + 内容（非 md 用带语言代码块）；**不**再收到文件。
3. `-f demo.rs`（>3000 字符）→ 收到 `demo.rs.docx`：顶部文件名 H1 + 等宽代码块（无高亮，保留缩进）。
4. 关闭开关①：短文本不再内联，按开关②决定转 docx 或发源文件。
5. 关闭开关②：文本文件一律发源文件（恢复现状）。
6. 非文本类（图片、pdf、xlsx、zip…）行为与现状一致。
7. 其它渠道（Telegram）对同一组 `-f` 文件仍发送源文件。
8. 转换失败时静默退回发送源文件，不影响其它文件与提问流程。

### 2026-06-06 实现联调反馈

1. **投递时序**：钉钉 `oToMessages/batchSend`（文本/markdown/文件 内联）为**异步投递**（HTTP 200 仅代表受理，落到聊天有队列延迟），而互动卡片 `createAndDeliver` 投递更快，会**插队**早于内联/附件到达，导致问题把内联内容顶上去、默认看不见。**修复**：在 message（正文+附件）全部发完后、发第一道题前等待 **1000ms**，让 batchSend 内容先落地（仅钉钉渠道，常量 `MESSAGE_SETTLE_DELAY`）。

2. **docx 标题字号（重要，覆盖 D14/计划§5 的标题字号取值）**：实测钉钉 docx 渲染器对字号**渲染不可靠**：
   - **命名标题样式（Heading1/2/3）、或带 `keepNext` 的段落**，其字号会被钉钉**抹平**成同一档；
   - **相近字号会倒置**（如 sz30 显示比 sz36 大、sz32 比 sz40 大），中英文一致、可复现；
   - 字号映射呈"锯齿"：`sz 20→34` 干净单调递增，**`sz≈36` 处塌缩**（显示仅约 20 大小），之后再递增。
   - **因此标题不再用命名样式**，改为「**普通段落 + run 上直接写 加粗 + 字号**」。实测可单调渲染的取值：**H1=56(28pt)、H2=32(16pt)、H3 及更深=28(14pt)、正文=24(12pt)**（均落在干净单调带，H1 用高位大字）。H1/H2 保留浅灰下边框（`D8DEE4`）作 GitHub 风格下划线；加大段前距（before：H1=640 / H2=600 / H3=480 twips），避免标题与上文贴太紧。H4–H6 退化到 H3。
   - 说明：原型阶段样张只含 H1/H2（48/36），恰好渲染正常故未暴露；真实文档引入 H3 后才发现上述问题。
