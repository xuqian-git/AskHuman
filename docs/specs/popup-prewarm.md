# 需求：弹窗预热（方案6 —— 进程池压「WebView/页面加载」大头）

> 状态：设计已与用户对齐（2026-06，访谈两轮 + 机制确认），待计划评审后实施。
> 关联：`docs/specs/popup-launch-performance.md`（§4 方案6、§6.3 预热分析、§7 度量方法论）。
> 实现计划：`docs/plans/popup-prewarm.md`。

## 1. 背景与目标

经方案1/2/3/4/5 后，弹窗端到端 WARM p90 ≈ 520ms 中 **`GUI show→painted ≈ 496ms`** 占 ~92%，几乎全是
**每弹窗一进程的 WebView 原生初始化 + 页面加载 + Vue 挂载**——单次调用内无法与自身并行的硬成本
（`window visible ~250ms` + `page boot ~435ms`）。要再大幅压，只能把这块从关键路径「藏掉」= **预热**：
维持一个已初始化好、已挂载、隐藏待命的弹窗，来请求时只「注入内容 + show()」（目标把 `show→painted`
降到数十 ms 量级）。

**目标**：在 daemon 已运行的稳态下，让弹窗从「调用 → 看到内容」尽量逼近「IPC + 注入渲染」量级，
消除 WebView/页面加载在关键路径上的占用。

**非目标**：
- 不加速 **daemon 冷启动后的第一个弹窗**（见 §3 D-M1）。
- 不改 stdout 契约、退出码、结果区块格式、抢答语义、作答回传路径。
- 不涉及非 Unix（非 Unix 仍走单进程回退，方案6 自动不生效）。

## 2. 总体形态（用户已确认）

**daemon 维护一个「热弹窗 helper 进程」池**：预先起好一个 `AskHuman --popup`（复用现有短命弹窗进程），
让它**提前完成 WebView 初始化 + 载入 index.html + 挂载 Vue/App.vue**，停在 **PopupView 无请求待命态**
（窗口隐藏）。来请求时，daemon **不再新 spawn 进程**，而是把该请求的 `Show` 喂给池中这个热进程，
热进程把内容注入已挂载的前端、绘制完成后 `show()` 上屏。热进程**答完即退**（沿用现有模型），
daemon 随即后台再补一个，恒维持池中有 1 个就绪热进程。

**不复用 gui-host**：gui-host 是 Unix-only、仅 `menuBarIcon != off` 时常驻、生命周期最复杂（托盘/drain/
换新/accessory）。进程池方案复用现有 helper 代码、与托盘解耦、跨 unix 一致、风险最低、且天然单次使用
（杜绝「旧内容闪现」，见 D-M2）。

## 3. 设计决策（访谈结论）

- **D-Q1 承载方式**：daemon 预热 `--popup` helper 进程池（非 gui-host、非新角色进程）。
- **D-Q2 开关与默认**：**默认开**；提供配置开关可关闭（**不放实验性 Tab**，是常规通用项）。代价是常驻
  一个隐藏 WebView 进程（内存数十~上百 MB），用户可关。
- **D-Q3 池大小 / 并发**：恒维持 **1 个**就绪热实例；**并发**的第 2+ 个在途请求回退**冷 spawn**；热实例被
  领用或死亡后 daemon **后台重建**下一个。
- **D-Q4 回退与正确性**：方案6 是叠加在现有冷路径之上的**快路径**；热实例不就绪 / 无显示 / 出错一律
  **透明回退**到现有 `spawn_gui_helper` 冷路径，**绝不移除冷路径**。
- **D-Q5 预热深度**：预热到 **Vue/App.vue 挂载完成、PopupView 无请求待命**（预付 WebView init + 页面
  加载 + mount）；来请求只「注入内容 → 渲染 → show」。需前端支持「无请求挂载、后注入请求」。
- **D-M1 冷启动范围**：方案6 **不加速 daemon (重)启动后的第一个弹窗**——那一刻还没有就绪热实例，且
  预热本身需 ~500ms，来不及给正在等待的首个请求用。daemon 启动后**立即后台补热实例**，故**第一个之后
  全部走热路径**。daemon 长命 → 「第一个冷」很罕见。冷启动残余（daemon 进程创建 + 一次冷 WebView）
  不在本方案范围（方案3 已消除冷启动的 IM 阻塞）。
- **D-M2 杜绝「旧/空内容闪现」**（用户强需求）：靠两条保证——(1) 每个热实例**单次使用** + 挂载时
  **完全无请求**（空白）+ 用后即弃换新（不跨请求复用同一窗口，结构上没有「上一个提问」可残留）；
  (2) 窗口**保持隐藏，直到注入内容（DOM 已是正文）才 `show()`**——show 时合成的首帧即正文，无「加载中→正文」过渡。
- **D-M2′ 背景节流 / 隐藏窗 rAF（实现期修正，已实测）**：原计划「隐藏窗双 `rAF` 绘制完成才 `show()`」**不可行**。
  实测：窗口 ordered-out（从未 show）时**没有 display link 驱动帧 → rAF 根本不回调**；`background_throttling(Disabled)`
  只阻止**已合成视图**被节流，无法为隐藏窗驱动帧。故「先双 rAF 再 show」会让热进程**永不上屏**。
  **改法**：领用时前端 `nextTick` 等 Vue 把正文写入 DOM 后，**直接调后端 `popup_show_window` 上屏**（show 走原生
  调用、**不依赖 rAF**）；`fe.painted` 打点与 harness autodismiss 放到 **show 之后**的双 rAF（窗口可见后 rAF 恢复）。
  仍对**冷热弹窗都设 `background_throttling(Disabled)`**（缓解被遮挡时的节流）。
  **锁屏/息屏行为（实测）**：弹窗**照常上屏**（`gui.win_show` 触发，因 show 不依赖 rAF）；仅 show 后的 `fe.painted` /
  harness autodismiss 因无显示刷新而暂停——只影响 harness（其本就拒绝锁屏运行），不影响产品的可见上屏。
- **D-M3 无显示兜底**（因默认开）：headless / 无 DISPLAY（如纯 IM 的 Linux 服务器）**自动跳过预热**
  （零浪费、不因无显示而失败）；有可用显示才热。
- **D-M4 生命周期**：热实例连接**不给 daemon 续命**（与托盘订阅同策略，不计入 `active`、不引用于空闲
  判定）；daemon **空闲退出 / 检测到新二进制(进入 drain)** 时**回收**热实例，新 daemon / 新二进制起来后
  再补热。**不改既有空闲退出语义**。
- **D-M5 度量**：harness 增加 **`hot`** 档（daemon 温 + 已有就绪热实例），与 cold/warm 并列三档；为热
  路径补埋点，量化 `show→painted` 降幅与端到端收益。

## 4. 验收

1. daemon 已运行、热实例就绪时，`AskHuman` 弹窗 `show→painted` 大幅下降（目标数十 ms 量级），端到端
   p90 较 WARM 基线显著改善（具体阈值以 harness `hot` 档实测为准）。
2. **零「旧/空内容闪现」**：任何情况下用户首次看到窗口时即是**本次**提问的完整内容（不出现空白或上次内容）。
3. **正确性回退**：并发请求、热实例未就绪、无显示、热进程崩溃等场景，均**透明回退**冷路径，行为与方案6
   关闭时一致；作答/取消/抢答/配置热切/聚焦/Dock 角标/提示音/出现动画**全部不回归**。
4. **生命周期不回归**：daemon 仍能按既有规则空闲退出；graceful-drain 换新不被热实例阻塞；二进制换新后
   新热实例为新二进制。
5. 开关默认开、可关；关闭后完全等价于方案6 之前的行为。
6. 非 Unix / 无显示环境：方案6 不生效，行为不变。
