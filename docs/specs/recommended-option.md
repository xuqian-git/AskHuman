# 需求：推荐选项显式声明（`-o!`）

> 状态：已实现并覆盖 Popup、历史与四个 IM 渠道。
> 关联计划：`docs/plans/recommended-option.md`
> 影响面：CLI 入参（`cli/args.rs`、`cli/mod.rs`、`cli/help.rs`）、数据模型（`models.rs` + TS 类型）、弹窗 / 历史详情 UI、四个 IM 渠道的选项展示、i18n、参考提示词（`prompts.rs`）。**不改** stdout 结果区块契约、退出码语义、daemon 协议结构（仅 `Question` 内字段演进且向后兼容）。

## 1. 背景

参考提示词要求模型提问时「include your recommended answer」，但 CLI 没有结构化通道，模型只能把「（推荐）」写进 `-o` 的选项文字里。这导致：

- 推荐标记混进选项文本，用户提交后 `[选择的选项]` 区块会回传带「（推荐）」字样的脏文本；
- 各端无法对推荐选项做统一、明显的视觉呈现。

本需求新增显式语法：模型用 `-o!` 声明某个预定义选项是其推荐答案，各端以 UI 化的方式标注，提交值保持原文。

## 2. 目标

```bash
AskHuman "继续部署吗？" -o! "继续" -o "停止"
```

- 弹窗：「继续」选项文本前显示 UI 风格 Badge（大拇指图标 +「推荐」文字）；
- IM 渠道：「继续」的显示文本带「👍推荐 」前缀；
- 用户勾选后输出 `[选择的选项]` 仍是原文「继续」。

## 3. 已确认决策

| 编号 | 决策项 | 结论 |
|---|---|---|
| D1 | 语法 | 新增 `-o!`（长形式 `--option!`）：语义与 `-o` 完全一致（归最近声明的 `-q`；无 `-q` 时归被提升的问题），且把该选项标记为推荐。曾对比过后置 `--recommended`、变体 `-O`、末尾引用 `--recommend <文本>`，均不及 `-o!`（单 token 原子绑定、无错位/漏写面、无大小写混淆、无文本重复） |
| D2 | 前后缀对模型的影响 | 经分析无实质区别：两者决策粒度都是「逐选项、写该选项时」，后置仅多看到模型自己即将输出的文本，不构成新信息；`-o!` 反而漏写/错位面更小、更省 token。结论：采用前置 `-o!` |
| D3 | 数量 | 每题允许**多个**推荐选项；`-o!` 多次出现不报错 |
| D4 | 预选中 | 推荐选项**不预选中**，仅视觉标注（避免用户未看清就误提交） |
| D5 | 弹窗呈现 | 选项文本**前**放 UI 风格 Badge：SVG 大拇指图标（**非 emoji**）+ 本地化「推荐」文字（zh「推荐」/ en「Recommended」），样式为 accent 色调 pill，让它看起来是 UI 元素而非答案标题的一部分；历史详情页同样还原 |
| D6 | IM 呈现 | 选项**显示文本** = 本地化前缀「👍推荐 」（en「👍Recommended 」，随界面语言）+ 原文；**提交值仍为原文**（`selected_options` 不含前缀） |
| D7 | 数据模型 | `Question.predefined_options` 由 `Vec<String>` 改为对象数组 `Vec<OptionItem>`（`{text, recommended}`），未来可继续扩展选项属性；**反序列化兼容纯字符串**（旧 history.jsonl / 旧 IPC 负载读出即 `recommended=false`，零迁移） |
| D8 | 帮助与提示词 | `--agent-help` 增加 `-o!` 说明与示例（示例保持 `-o!` 与值之间留空格，规避交互式 shell 历史展开）；`prompts.rs` 参考提示词把「include your recommended answer」改为指引用 `-o!` 标记推荐（已安装的 Agent rules 需重新安装方可生效） |
| D9 | 输出契约 | `[选择的选项]` 等结果区块、退出码、stdout 洁净全部不变；推荐信息不出现在输出中 |

## 4. 约束与既有规则（不可破坏）

- **提交值为原文**：所有渠道（弹窗 / Telegram / 钉钉 / 飞书 / Slack，含各文本回退模式）回传的 `selected_options` 必须是不带前缀的原始选项文本。
- **历史兼容**：旧 `history.jsonl`（选项为字符串数组）必须能正常读出并在历史窗口还原展示。
- **既有入参语义不变**：`-o` / `-q` / `-f` / `--stdin` / `--no-markdown` 行为保持；`-o!` 的归属与报错规则（如不能出现在第一个 `-q` 之前）与 `-o` 一致。
- 解析逻辑保持纯函数可单测。

## 5. 验收标准

1. `AskHuman "Q" -o! "A" -o "B"`：弹窗中 A 前出现「大拇指 + 推荐」Badge，不预选中；勾选 A 提交后输出 `[选择的选项]` 为 `A`（无前缀）。
2. `-q` 多题中 `-o!` 归属正确；一题内多个 `-o!` 均被标注。
3. `--option!` 与 `-o!` 等价；`-o!` 出现在第一个 `-q` 之前（有 `-q` 时）→ 报错退 1（与 `-o` 一致）。
4. 四个 IM 渠道：推荐选项显示文本带本地化「👍推荐 」前缀；勾选提交后回传原文。钉钉卡片回调（模板回传显示文本）能正确还原为原文。
5. 各渠道文本回退模式：编号清单中推荐选项行带前缀；按编号回复仍映射到原文。
6. 旧历史记录正常加载；新记录在历史详情中还原推荐 Badge。
7. `--agent-help` 含 `-o!` 说明与示例；`prompts.rs` 参考提示词更新。
8. 既有用法（无 `-o!`）回归正常；输出区块与退出码不变。
