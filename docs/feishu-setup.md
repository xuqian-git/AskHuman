# 飞书 / Lark 渠道配置指南

本文档说明如何在飞书（或国际版 Lark）开放平台创建并配置一个企业自建应用，使 HumanInLoop 的「飞书 Channel」可用。该渠道采用 **企业自建应用 + 机器人 + 长连接（WebSocket）+ 单聊** 形态，**无需公网**即可收发消息与卡片回调。

> 国际版 Lark 与飞书国内步骤一致，区别仅在域名：开放平台为 `open.larksuite.com`，HumanInLoop 设置页「服务域名」填 `https://open.larksuite.com`（国内留空即用默认 `https://open.feishu.cn`）。

## 一、创建应用并启用机器人

1. 进入飞书开放平台 → 开发者后台 → 创建「企业自建应用」。
2. 记录应用凭证：**App ID**（`cli_...`）与 **App Secret**（填入 HumanInLoop 设置页）。
3. 「应用能力 → 添加应用能力」中启用 **机器人（Bot）**。

## 二、导入权限（Scopes）

在「权限管理 → 开通权限」中，可使用「批量导入」并粘贴本仓库的 [`docs/feishu-permissions.json`](./feishu-permissions.json)：

```json
{
  "scopes": {
    "tenant": [
      "im:message:send_as_bot",
      "im:message.p2p_msg:readonly",
      "im:message:readonly",
      "im:resource"
    ],
    "user": []
  }
}
```

各权限对应的功能（按本渠道实际调用的接口逐项核对官方权限要求）：

| 权限 | 用途 | 涉及接口 |
| --- | --- | --- |
| `im:message:send_as_bot`（以应用身份发消息） | 给用户单聊发送 文本 / 图片 / 文件 / 互动卡片，并在收尾时 PATCH 更新卡片 | 发送消息、更新应用发送的消息卡片 |
| `im:message.p2p_msg:readonly`（读取用户发给机器人的单聊消息） | 长连接接收用户在单聊中发来的消息事件 | 事件 `im.message.receive_v1` |
| `im:message:readonly`（获取单聊、群组消息） | 下载用户发来的图片 / 文件资源（人 → AI 附件回传） | 获取消息中的资源文件 |
| `im:resource`（获取与上传图片或文件资源） | 发送 `-f` 附件前先上传媒体取 key | 上传图片、上传文件 |

说明：

- 内部自建应用使用 `tenant_access_token`，`user` 权限留空即可。
- `im:resource` 为「高级权限」，部分企业可能需要管理员审批后才能生效。
- 若你不需要「AI → 人发文件 / 人 → AI 回传图片文件」，可不申请 `im:resource` 与 `im:message:readonly`；但会丧失对应能力，建议保留以与钉钉 / Telegram 对齐。

## 三、事件与回调（非权限项，但必须配置）

权限只是「能调哪些接口」，要让机器人能**收到**消息和卡片提交，还需在后台配置。
**关键：飞书把「事件」和「回调」放在两个独立的配置页，必须分别开启长连接——只配其一，会出现「能收消息但卡片点提交无反应/转圈回弹」或「收不到消息」。**

> ⚠️ 两个页面在切换为「长连接」并保存时，都要求**本地已有一条该应用的长连接在线**。做法：先让 HumanInLoop 这端建好连接（设置页点「自动识别」会建连并保持约 120s，或正处于一次提问等待中），再去后台点保存。

### 3.1 事件配置（收消息）

1. 进入「开发配置 → 事件与回调 → **事件配置**」。
2. 编辑订阅方式 → 选择「**使用长连接接收事件**」→ 保存。
3. 在「已添加事件」中**添加事件「接收消息 `im.message.receive_v1`」**：用于接收用户在单聊发送的文字 / 图片 / 文件。
   > ⚠️ 据飞书官方 [机器人常见问题](https://open.feishu.cn/document/faq/bot)，机器人**必须订阅「接收消息」事件并发布后，单聊窗口才会出现输入框**。否则打开机器人是「白屏 / 没有输入框」，也无法用「自动识别」发码——最常见踩坑点。

### 3.2 回调配置（卡片提交）

1. 进入「开发配置 → 事件与回调 → **回调配置**」（与事件配置**不是同一个页面**）。
2. 编辑订阅方式 → 选择「**使用长连接接收回调**」→ 保存。
3. 在「已订阅的回调」中**添加回调「卡片回传交互 `card.action.trigger`」**：用户点卡片「提交」时，勾选项 + 补充文字经此回调回传。
   > ⚠️ 必须用**新版** `card.action.trigger`；「消息卡片回传交互（旧）`card.action.trigger_v1`」**不支持长连接**。
   > 若漏配本节，卡片能收到但点「提交」会**转圈几秒后回弹、提交不成功**。

## 四、发布与可用范围

1. 在「版本管理与发布」中创建版本并发布（权限、机器人能力、事件订阅均需发布后生效）。
2. 确认机器人的**可用范围**包含目标用户，否则发送会报 `230013 Bot has NO availability to this user`。

## 五、在 HumanInLoop 中填写

打开 HumanInLoop 设置页 → 「通信渠道」→「飞书」，开启开关后填写：

| 字段 | 说明 |
| --- | --- |
| App ID | 应用凭证 `cli_...` |
| App Secret | 应用凭证密钥 |
| Open ID | 接收 / 作答用户的 Open ID（单聊）。可点「自动识别」：先校验 App ID/Secret，再提示你用目标账号私聊机器人发送一个 4 位数字以精确回填。**前提**：已按上一节订阅「接收消息」事件并发布，否则机器人单聊没有输入框，无法发码（见常见问题） |
| 服务域名 | 留空 = 飞书国内 `https://open.feishu.cn`；Lark 国际版填 `https://open.larksuite.com` |

填好后点「测试连接」：会换取 token 并向该 Open ID 的单聊发送一条测试消息，飞书内收到即配置成功。

## 六、交互与回退行为

- 提问以**互动卡片**（卡片 JSON 2.0，直接下发无需搭建模板）逐题发送：卡片表单内勾选预定义选项（多选）、可补充文字，点「提交」完成该题（回调走长连接，须 3 秒内回包，已自动处理）。
- 提交后卡片**复刻钉钉「已提交」态**：同一张卡片保留题目与选项，勾选器禁用但保留你的勾选、补充文字回显在输入框、按钮由「提交」变为禁用的「已提交」（而非换成一行状态文字）。
- 作答期间在聊天里发送的**图片 / 文件**会被累积进该题答案；**纯文字会被忽略**（请用卡片输入框补充文字）。
- 若卡片投放失败，会自动**回退**为「纯文本 + 编号选项」：回复编号（多选用逗号，如 `1,3`）/ 直接输入文字 / 发送图片 / 文件均可作答。
- 多渠道同启时以「整个会话」为粒度**抢答**：哪端先答完全部题即采用该端结果，其余自动收尾（飞书卡片会被 PATCH 成「已在 X 回答」终态）。

## 七、常见问题

| 现象 | 可能原因 |
| --- | --- |
| **机器人单聊白屏 / 没有聊天页面 / 没有输入框** | 未订阅「接收消息」事件（`im.message.receive_v1`，需 `im:message.p2p_msg:readonly`）或未发布。据官方 FAQ，订阅该事件并发布后单聊才会出现输入框 |
| `230006 / 234007 Bot ability is not activated` | 未启用机器人能力，或发布后未生效 |
| `230013 Bot has NO availability to this user` | 目标用户不在机器人可用范围内 |
| 发送 / 上传报权限错误 | 对应 scope 未开通或未发布；`im:resource` 可能待管理员审批 |
| 卡片能收到但点「提交」转圈后回弹、提交不成功 | **「回调配置」页**未把订阅方式设为长连接、或未添加「卡片回传交互 `card.action.trigger`」（见 §3.2）。注意它与「事件配置」是两个独立页面，常见漏配 |
| 收不到用户消息 / 单聊没有输入框 | **「事件配置」页**未订阅 `im.message.receive_v1`、订阅方式不是长连接、或未发布（见 §3.1） |
| 想确认回调是否到达 | 设环境变量 `HUMANINLOOP_FEISHU_DEBUG=1` 运行，点提交后查看 `~/.humaninloop/feishu-debug.log`：有 `event_type=card.action.trigger` 即回调已到达 |
| 测试连接失败 | App ID / App Secret 错误，或服务域名填错（国内 vs 国际版） |
