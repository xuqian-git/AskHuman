# 待办 / 已知问题

## 已知问题（钉钉渠道）

### 1. 同一 client-id 同一时刻仅允许一条 Stream（多开相互干扰）

- **根因**：钉钉官方限制——同一个 client-id 同一时间只允许启动一条 Stream 服务，多开会相互干扰（见官方排查清单）。
- **现状**：当前每次 `AskHuman` 进程各自开一条 Stream（同一 client-id）。正常「一次只问一题」无碍，但**连续快速 / 并发提问时**多条 Stream 会抢消息，可能把用户回复投递到错误的连接。
- **候选修复**：
  1. 文件锁串行化——同一时刻只允许一个进程持有 Stream，退出时发 Close 帧干净断连再释放锁（轻量、无常驻进程；每次提问需重新建连，并发会排队）。
  2. 常驻 daemon——后台进程独占持有 Stream，各 `AskHuman` 经本地 socket 注册等待、由其转发用户消息（真复用 / 连接常热 / 支持并发，但需管理 daemon 生命周期 + IPC，复杂度高）。
- **状态**：暂不修，已记录（另见 `docs/specs/dingtalk-channel.md` 末尾反馈意见）。

### 2. 机器人发送「钉钉不支持类型」的文件无法可靠预览

- **背景**：机器人只能用 `media/upload` + `sampleFile` 发文件；`sampleFile` 的 `fileType` 仅支持 `doc/docx/xls/xlsx/ppt/pptx/zip/pdf/rar`；且机器人媒体文件**不进钉盘**，不像用户手动上传的钉盘文件那样可内嵌预览。
- **实测（2026-06-04）**：
  | 发送方式 | 钉钉表现 |
  |---|---|
  | `fileType=pdf`（受支持） | 直接预览 / 打开 ✅ |
  | `fileType=md`（不支持） | 下载被当成 `.file` 后缀，打不开 ❌ |
  | `fileType=txt`（UTF-8 无 BOM） | 能打开但乱码（疑似按 GBK 解码）❌ |
  | `fileType=txt`（UTF-8 +BOM） | 编码正确但文字显示为白色、看不见（拷出可见）❌ |
  | `fileType=html` | 也不行 ❌ |
- **影响**：md / txt / log / html / 代码等常见文本类附件无法在钉钉内可靠预览。
- **候选方案（待定）**：
  1. 文本类（md/txt/log/代码）转 **PDF** 发送（已验证 PDF 预览稳定；md 转纯文本 PDF 则语法不渲染）。
  2. 文本类直接贴成 **Markdown 消息**（钉钉原生渲染，但非可下载文件）。
  3. 一律打包成 **zip**（可下载，但不预览）。
- **状态**：暂不处理，已记录。
